# fuxi 技术架构（v1）

> 状态：已确认  
> 适用阶段：MVP v1  
> 产品定位：长期自用的本地 AI 知识编译系统  
> 产品定义：[07-ai-product-thinking-20260604.md](./07-ai-product-thinking-20260604.md)  
> 架构依据：[08-ai-architecture-thinking-20260610.md](./08-ai-architecture-thinking-20260610.md)

## 1. 技术目标

用最少的技术组件跑通以下闭环：

```text
链接 / PDF / 想法
    ↓
保存原料
    ↓
生成单源笔记
    ↓
添加固定标签和关联
    ↓
更新主题页
    ↓
问答 / 查找 / 生成选题
```

选型遵循四项原则：

1. AI 熟悉：优先 Python、Markdown、Git 等主流工具。
2. 本地运行：程序和文件保存在本机，不部署服务器。
3. 文件优先：Markdown 是主数据，不使用数据库。
4. 社区成熟：只选择文档完整、社区活跃的工具。

## 2. 最终技术栈

| 层级 | 技术 | 用途 |
|---|---|---|
| 运行环境 | Ubuntu + Python 3.12 | 运行本地机械处理脚本 |
| 操作入口 | Codex 会话 + 项目 Skill | 通过自然语言执行入库、编译、问答和选题工作流 |
| 主数据 | Markdown | 保存笔记、主题页、索引和选题 |
| 元数据 | YAML frontmatter | 保存来源、标签、关联关系等结构化字段 |
| 普通网页提取 | Trafilatura | 将网页正文提取为 Markdown |
| PDF 提取 | `pypdf` | 提取文字型 PDF 的文本 |
| 索引 | `INDEX.md` | 为 Codex 和用户提供知识导航 |
| 关联 | 固定标签词表 + `related` 字段 | 建立可解释的双向关联 |
| 版本管理 | 本地 Git | 查看差异，并在主题页重写错误时回退 |
| AI 模型 | Codex | 摘要、判断、标签、冲突识别和主题编译 |

## 3. 进：输入层

### 3.1 支持的输入

| 输入 | v1 处理方式 | 失败处理 |
|---|---|---|
| 普通网页链接 | Trafilatura 提取正文并保存到 `raw/` | 手动复制正文并保存为 Markdown |
| GitHub 公开页面 | 先按普通网页处理 | 必要时再引入 `gh` CLI，不作为 v1 必需项 |
| 文字型 PDF | 使用 `pypdf` 提取文本，原 PDF 保留在 `raw/` | 手动复制可读文字 |
| 扫描件或图片型 PDF | v1 不自动识别 | 手动录入，不引入 OCR |
| 微信、X 等反爬来源 | 手动保存为 Markdown 或 PDF | 不维护平台专用爬虫 |
| 个人想法 | 直接写入 `inbox/` | 无 |

### 3.2 输入原则

- 所有输入先落盘，再进行 AI 加工。
- 原始文件只读保存，不被摘要或主题页覆盖。
- 可以把内容发送给 Codex 处理。
- 密钥、隐私资料和受保密约束的文档不得发送给云端模型。

### 3.3 更简单的替代

最初 10 至 20 条素材可以完全由 Codex 手动读取和写文件，不先开发抓取脚本。只有重复操作已经形成稳定模式后，才增加自动提取。

## 4. 存：数据层

### 4.1 目录结构

```text
项目根目录/
├── AGENTS.md
├── README.md
├── inbox/          # 等待处理的链接、PDF和想法
├── raw/            # 原始网页、PDF及提取文本，只读
├── notes/          # 单源笔记
├── wiki/           # 主题页和实体页
├── briefs/         # 选题
├── indexes/        # 自动生成的索引
├── templates/      # Markdown 模板和固定标签词表
├── scripts/        # 第二阶段增加的机械处理脚本
└── tests/          # 脚本测试
```

### 4.2 数据分层

| 目录 | 内容 | 修改规则 |
|---|---|---|
| `inbox/` | 尚未处理的输入 | 处理完成后标记或归档 |
| `raw/` | 原文、原始 PDF、提取文本 | 只读，不覆盖 |
| `notes/` | 每项素材对应的单源笔记 | 可以补充，不代替原文 |
| `wiki/` | 多篇笔记编译形成的主题页和实体页 | 新资料进入后允许重写 |
| `briefs/` | 基于主题页生成的选题 | 按需生成 |
| `indexes/` | 文件、标签和主题导航 | 由脚本或 Codex 生成 |

### 4.3 Markdown 元数据

v1 只保留必要字段：

```yaml
---
id: ""
title: ""
type: note
source: ""
created_at: ""
tags: []
related: []
---
```

约束：

- `id` 创建后不得修改。
- `type` 使用固定枚举，例如 `note`、`topic`、`entity`、`brief`。
- `tags` 只能从固定标签词表中选择。
- `related` 保存相对路径，并保持双向关联。
- 标签词表保存在 `templates/`，新增标签前先更新词表。

### 4.4 索引与关联

v1 不使用向量库和 embedding。

关联生成顺序：

1. Codex 根据内容从固定词表中选择标签。
2. 比较标签重叠和标题关键词。
3. 将达到阈值的关联写入双方的 `related` 字段。
4. 更新对应的主题页和索引。

这种方式可解释、可通过 Git 查看差异，也不需要额外服务。

### 4.5 版本回退

- 使用本地 Git 管理 Markdown 和脚本。
- 重写 `wiki/` 中的主题页前，必须确保现有内容已进入 Git 历史。
- 原始资料不因重新编译而改变。
- Git 初始化属于实施前置条件；当前文档所在目录尚未初始化为 Git 仓库。

## 5. 取：输出层

### 5.1 问答

Codex 按以下顺序读取：

1. `indexes/INDEX.md`
2. 相关的 `wiki/` 主题页或实体页
3. 必要时读取 `notes/`
4. 对事实存在疑问时回查 `raw/`

优先使用已经编译的主题页，不在每次问答时重新遍历所有原文。

### 5.2 查找

v1 使用文件名、标题、标签和关键词搜索，不建立独立搜索服务。内容增长后再增加一个 Python 搜索脚本。

### 5.3 选题

Codex 读取相关主题页，把可执行选题写入 `briefs/`。每个选题应包含：

- 选题标题
- 核心观点
- 可引用的笔记和原料
- 当前资料缺口

### 5.4 周报

周报不属于 v1。v2 再增加手动触发的周报生成，不在当前阶段引入定时任务。

### 5.5 用户界面

这是长期自用产品，v1 不开发图形界面或本地网页。主要通过 Codex 会话操作，Markdown 阅读器仅作为可选外壳，不进入核心架构。

## 6. AI 与脚本的职责边界

### Codex 负责

- 阅读和理解原始内容
- 编写带判断的单源笔记
- 从固定词表中选择标签
- 判断资料之间的语义关联
- 识别新旧资料的矛盾
- 重写主题页和实体页
- 回答问题和生成选题

### Python 负责

- 提取网页和 PDF 文本
- 扫描文件及解析 frontmatter
- 校验必填字段、标签和失效链接
- 计算简单的关联分数
- 生成索引

原则：机械、确定的操作交给脚本；需要理解和判断的操作交给 Codex。

## 7. 分阶段实施

### 阶段一：零代码验证

只建立：

1. 分层目录
2. `AGENTS.md`
3. 项目 Skill
4. Markdown 模板
5. 固定标签词表
6. `INDEX.md`
7. 本地 Git

用 10 至 20 条真实资料手动跑通完整闭环。

### 阶段二：机械步骤自动化

确认工作流稳定后，只增加：

```text
scripts/
├── capture.py        # 提取网页和文字型 PDF
├── rebuild_index.py  # 重建索引
└── validate.py       # 校验元数据和关联
```

不为一次性操作提前编写抽象框架。

## 8. 明确不使用

v1 不使用以下技术：

- MySQL、PostgreSQL、MongoDB、SQLite
- Chroma、Milvus、Pinecone 等向量库
- LangChain、LlamaIndex 等 RAG 框架
- React、Next.js、Streamlit
- FastAPI 或前后端分离架构
- Docker
- 云服务器
- 定时任务
- OCR
- 多端同步和多人协作

## 9. 验收标准

v1 完成需要同时满足：

- 能录入普通网页、文字型 PDF 和个人想法。
- 抓取失败或遇到反爬时，可以手动保存后继续处理。
- 原始资料、单源笔记和编译结果分层保存。
- Codex 能根据固定标签建立关联并更新主题页。
- 主题页重写前的版本可以通过 Git 恢复。
- 能通过 Codex 完成问答、查找和生成选题。
- 使用 10 至 20 条真实资料连续验证完整流程。
- 没有引入数据库、向量库、服务器或独立 UI。

## 10. 已确认决策

| 决策项 | 结论 |
|---|---|
| PDF | v1 支持文字型 PDF，不支持 OCR |
| 微信、X 等来源 | 接受手动保存 |
| 云端 AI | 允许把非敏感内容发送给 Codex |
| 主题页重写 | 接受 Git 留档后重写 |
| 标签 | 使用固定标签词表 |
| 周报 | 放到 v2 |
| 使用对象 | 长期自用，不开发独立 UI |
