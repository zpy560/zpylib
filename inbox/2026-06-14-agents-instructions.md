## Coding 的规范和原则

### 1. 编码前思考

**不要假设。不要隐藏困惑。呈现权衡。**

在 coding 实施前：

- **明确说明假设**：如果不确定，询问而不是猜测。
- **呈现多种解释**：当存在歧义时，不要默默选择。
- **适时提出异议**：如果存在更简单的方法，说出来。
- **困惑时停下来**：指出不清楚的地方并要求澄清。

### 2. 简洁优先

**用最少的代码解决问题。不要过度推测。**

对抗过度工程的倾向：

- 不要添加要求之外的功能。
- 不要为一次性代码创建抽象。
- 不要添加未要求的“灵活性”或“可配置性”。
- 不要为不可能发生的场景做错误处理。
- 如果 200 行代码可以写成 50 行，重写它。

**检验标准：** 资深工程师会觉得这过于复杂吗？如果是，简化。

### 3. 精准修改

**只碰必须碰的。只清理自己造成的混乱。**

编辑现有代码时：

- 不要“改进”相邻的代码、注释或格式。
- 不要重构没坏的东西。
- 匹配现有风格，即使你更倾向于不同的写法。
- 如果注意到无关的死代码，提一下，不要删除它。

当改动产生孤儿代码时：

- 删除因本次改动而变得无用的导入、变量或函数。
- 不要删除预先存在的死代码，除非被要求。

**检验标准：** 每一行修改都应该能直接追溯到用户的请求。

### 4. 目标驱动执行

**定义成功标准。循环验证直到达成。**

将指令式任务转化为可验证的目标：

| 不要这样做 | 转化为 |
|---|---|
| “添加验证” | “为无效输入编写测试，然后让它们通过” |
| “修复 bug” | “编写重现 bug 的测试，然后让它通过” |
| “重构 X” | “确保重构前后测试都能通过” |

对于多步骤任务，说明一个简短的计划：

```text
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
3. [步骤] → 验证: [检查]
```

强有力的成功标准让 LLM 能够独立循环执行。弱标准（“让它工作”）需要不断澄清。

## 调试纪律

- 涉及外部 API、协议或 SDK 的问题，第一步找官方文档或可运行的参考实现（GitHub demo、SDK 源码），逐行对比请求头、请求体、参数生成逻辑，不要靠猜测和加日志摸索。
- 对比必须逐字段完整，包括 headers、body 结构、工具函数内部实现，“看起来差不多”不算对比过。
- 所有差异一次性修完再测试，不要改一个点就让用户验证，尤其是高成本操作（扫码、重新部署、重启服务）。
- 禁止无证据归因，“可能是 X 导致的”必须先验证再下结论，不要把猜测当方向让用户配合排查。
- 调试超过两轮未解决，立刻停下来换策略：找参考实现、搜索已知问题、对比工作版本和当前版本的 diff，而不是继续同一个方向加日志。

## 项目规则

### 项目目标

本项目是长期自用的本地 AI 知识编译系统。v1 只跑通：

```text
链接 / 文字型 PDF / 想法
→ 原料
→ 单源笔记
→ 标签与关联
→ 主题页
→ 问答 / 查找 / 选题
```

技术边界以 `techarc.md` 为准。修改实现边界前，先更新该文档。

### 目录约定

- `inbox/`：等待处理的链接、PDF 和想法。
- `raw/`：原始资料和提取文本。只追加，不覆盖。
- `notes/`：一项来源对应一篇单源笔记。
- `wiki/`：主题页和实体页，可随新资料重新编译。
- `briefs/`：基于主题页生成的选题。
- `indexes/`：生成式导航和索引。
- `templates/`：Markdown 模板、标签词表和数据约定。
- `scripts/`：确定性机械任务；v1 流程稳定前不新增脚本。
- `tests/`：脚本测试。
- `.agents/skills/`：本项目专用 Codex Skill。

不要在项目根目录外再创建 `fuxi/` 包装目录。

### 数据规则

1. Markdown 是主数据，不使用数据库或向量库。
2. `raw/` 中的原料是事实来源，禁止摘要覆盖原文。
3. 文件名使用 `YYYY-MM-DD-slug.md`；slug 使用小写英文和连字符。
4. 笔记元数据必须符合 `templates/frontmatter-schema.md`。
5. `tags` 只能从 `templates/tags.yaml` 中选择。
6. `related` 使用相对路径；建立关联时同步维护双向关系。
7. `id` 创建后不得修改。
8. 不把密钥、隐私资料或受保密约束的内容发送给 Codex。

### 工作流

#### 入库

1. 先把输入保存到 `inbox/`。
2. 把原始文件或正文保存到 `raw/`。
3. 使用模板生成 `notes/` 单源笔记。
4. 从固定词表选择标签，不临时发明标签。
5. 查找关联笔记，更新双方的 `related`。
6. 更新相关 `wiki/` 页面和 `indexes/INDEX.md`。
7. 完成后报告新增和修改的文件。

#### 问答

按以下顺序读取，够用即停：

1. `indexes/INDEX.md`
2. `wiki/`
3. `notes/`
4. `raw/`

回答需要区分原文事实、笔记判断和 AI 推断。

#### 选题

选题写入 `briefs/`，必须包含标题、核心观点、来源、资料缺口。

### 修改与安全

- 修改 `wiki/` 前先确认当前版本已进入 Git 历史；如果仓库尚无基线提交，先提醒用户。
- 删除文件、移动原料、Git 回滚、安装依赖前必须获得用户确认。
- 抓取失败或遇到反爬时停止自动尝试，改为提示用户手动保存。
- 文字型 PDF 属于 v1；扫描件和图片型 PDF 不做 OCR。
- 周报属于 v2，不添加定时任务。
- 不开发独立 UI，不引入 Docker、Web 服务或 RAG 框架。

### 验证

每次变更后至少检查：

1. 文件是否位于正确目录。
2. frontmatter 是否完整。
3. 标签是否来自固定词表。
4. `related` 是否为有效相对路径且双向一致。
5. 索引是否需要更新。
6. 是否意外修改了 `raw/` 原料。
