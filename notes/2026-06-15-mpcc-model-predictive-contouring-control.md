# MPCC：将路径进度纳入优化的自主赛车预测控制

## 一句话结论

该仓库是理解自主赛车 MPCC 的高价值研究参考实现：它把参考路径进度作为决策变量，在赛道约束内同时优化轮廓误差、滞后误差与前进速度，再通过逐次线性化和 HPIPM 求解时变 QP；但代码最后更新于 2021 年，依赖未锁定、测试未自动执行、C++ 不含避障且默认参数只适合小比例赛车，不能直接作为全尺寸车辆控制器。

## 原文要点

### MPCC 与普通轨迹跟踪 MPC 的关键差异

普通轨迹跟踪通常先给定带时间戳的参考状态，再最小化车辆状态与该参考的误差。MPCC 不强制车辆在某一时刻到达固定参考点，而是引入虚拟路径进度 `s` 和进度速度 `vs`：

```text
车辆状态 + 路径进度
→ 计算 contouring error 与 lag error
→ 在赛道边界和车辆约束内最大化进度
```

contouring error 近似车辆到参考线的法向距离，lag error 描述路径进度估计与车辆实际投影的切向偏差。前者保证不偏离赛道主线，后者保证进度状态仍代表车辆当前位置。该设计让控制器能够根据抓地力和曲率自行决定“追踪参考线的哪个位置、以多快速度前进”。

### C++ 优化变量

C++ 版本将驱动命令、转角和虚拟进度速度纳入状态，再将其变化率作为控制输入：

```text
x = [X, Y, phi, vx, vy, r, s, D, delta, vs]
u = [dD, dDelta, dVs]
```

这样输入变化率约束直接进入动力学，同时避免额外差分提升。默认预测周期为 20 ms、60 步，每次控制执行两轮 SQP。

### 从非线性问题到实时 QP

实现没有直接求解完整非线性规划，而是在当前预测轨迹附近：

1. 线性化并离散化动态自行车模型。
2. 一阶近似 contouring 和 lag error。
3. 用两个边界半空间近似赛道走廊。
4. 加入状态、输入、输入率、轮胎和软约束。
5. 归一化数值后交给 HPIPM。
6. 将新解与旧解按固定系数混合，再进入下一轮 SQP。

这是一种工程上常见的实时迭代思路，但当前实现没有 line search 或 merit function；固定混合和失败后重置初值属于研究代码中的简化。

### C++ 与 MATLAB 不是同一能力集合

| 维度 | C++ | MATLAB |
|---|---|---|
| 定位 | AMZ Driverless 启发的新版本 | 更接近原始 1:43 赛车论文 |
| 求解器 | 仅 HPIPM | HPIPM、YALMIP、CVX、quadprog |
| 轮胎约束 | 前轮侧偏角、后轮摩擦椭圆 | 基础论文形式 |
| 侧偏角代价 | 有 | 无对应 C++ 增强项 |
| 避障 | 无 | 一维 DP 选走廊 |
| 全尺寸模型 | `master` 不适用，另有分支 | 提供 FullSize 选项 |

因此不能笼统地说“仓库支持某能力”，必须指明语言与分支。

## 我的判断

### 最大价值是展示“规划与控制的中间形态”

MPCC 不是先规划完整速度轨迹再跟踪，也不是自由空间全局规划。它在已知赛道走廊和参考线条件下，将局部路径选择、速度推进与车辆稳定性放进同一个有限时域优化问题。对自动驾驶算法工程师，它最值得研究的不是 HPIPM API，而是：

- 如何用弧长参数化消除固定时间参考。
- 如何把几何误差变成可优化的二次近似。
- 如何从非线性轮胎模型构造逐次 QP。
- 如何用 track corridor、trust region 和软约束维持可解性。
- 如何通过 warm start 支撑高频闭环求解。

### 不能把赛车闭环直接迁移到道路自动驾驶

该问题预设赛道闭环、参考线已知、道路参与者稀少，并以进度最大化为核心。道路自动驾驶还需要：

- 路权、信号灯、限速和舒适性。
- 多主体预测与交互。
- 借道、换道、停车等离散决策。
- 定位和地图不确定性。
- 执行器延迟、故障处理和独立安全监督。

MPCC 更适合作为局部高动态控制内核或赛车研究基线，而不是完整 planning-control 系统。

### 数值稳定性比公式本身更决定可用性

源码已暴露多个必须工程化处理的问题：

- 低纵向速度时将 `vx` 强制抬到阈值，并清零部分状态，说明动态模型存在低速奇异性。
- 路径进度使用 trust region，说明线性化只在局部有效。
- 固定 SQP 混合缺少全局收敛保障。
- 多次求解失败后直接重置 warm start。
- 参数归一化是 HPIPM 数值表现的重要组成。

迁移时必须同时带走初值、尺度、失败恢复和模型有效域，不能只复制代价函数。

### 工程成熟度低于算法参考价值

- 默认分支最后提交时间为 2021-11-11。
- 安装脚本直接克隆未固定版本的五个依赖。
- 文档仍引用 Python 2.7。
- 测试函数默认全部注释。
- 没有 CI、独立库目标或稳定公共 API。
- `master` 与独立 `main` 分支并存。
- README 明确表示 C++ `master` 不适用于全尺寸车。

因此合理用法是固定提交、复现实验、抽取算法设计，再在自己的控制框架中重新实现和测试。

## 可复用内容

### 阅读顺序

1. 根目录 README：先理解 MPCC 数学问题和论文关系。
2. `C++/config.h`、`types.h`：确认状态、输入和预测维度。
3. `Spline/`：理解参考线弧长参数化和车辆投影。
4. `Cost/`：跟踪 contouring、lag、progress 和 beta 代价。
5. `Model/`：核对动态自行车模型、线性化和积分。
6. `Constraints/`：查看赛道、侧偏角与轮胎椭圆约束。
7. `MPC/mpc.cpp`：理解 warm start、SQP、归一化和失败恢复。
8. `Interfaces/hpipm_interface.cpp`：最后阅读 QP 数据布局。
9. MATLAB `getNewBorders.m`：单独研究 DP 避障走廊。

### 迁移到现代控制项目的最小清单

1. 固定 BLASFEO、HPIPM、Eigen 和 JSON 依赖版本。
2. 将控制器、车辆模型、赛道和可视化拆成独立目标。
3. 用 GoogleTest 或现有测试框架重写当前手动检查。
4. 为线性化 Jacobian 添加有限差分一致性测试。
5. 建立圆环、发卡弯、低速、饱和与不可行场景回归集。
6. 明确模型有效速度域和低速模型切换策略。
7. 增加 line search、merit function 或采用成熟 SQP/RTI 框架。
8. 将求解失败、约束松弛和控制降级纳入状态机。
9. 全尺寸车应从 `full-size` 分支和对应论文重新核对，而非只改 JSON 参数。

### 与 CppRobotics 的学习关系

CppRobotics 的 MPC 示例适合理解有限时域优化和跟踪控制的基本骨架；MPCC 进一步加入路径进度、动态轮胎约束、赛道走廊、SQP 和专用 OCP-QP 求解器。推荐先理解前者，再用本仓库研究高动态约束控制。

## 疑问与冲突

- 远端 `master` 是默认分支，但另有提交历史不同的 `main`，两者用途没有在 README 中解释。
- `full-size` 分支的具体 SQP 差异和当前可构建性本次未分析。
- C++ README 的 Python 2.7 说明与现代 CMake Python 查找方式可能不一致。
- 仓库测试是否能在固定历史依赖下通过，尚未实际验证。
- 默认模型、轮胎参数和求解时间是否可复现论文结果，需运行基准后确认。
