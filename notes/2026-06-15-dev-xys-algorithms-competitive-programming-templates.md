# Dev-XYS Algorithms：单文件 C++ 竞赛算法模板集

## 一句话结论

Dev-XYS Algorithms 是一个覆盖 103 种经典算法与数据结构的 C++ 单文件模板集，适合按名称快速查找图论、字符串、树结构、数论和排序实现；但项目自 2019 年后未更新，没有接口、文档、测试或 CI，全量 C++14 语法检查仍有两个文件失败，因此应把它当作学习和竞赛模板参考，而不是可信算法库。

## 原文要点

### 覆盖面广，定位偏信息学竞赛

仓库从基础排序、栈和队列覆盖到：

- Dijkstra、Tarjan、Dinic、ISAP、2-SAT。
- 线段树、树状数组、Splay、Treap、Link-Cut Tree。
- KMP、AC 自动机、后缀数组、后缀自动机、回文树。
- FFT、NTT、Miller-Rabin、原根、线性基。
- 可持久化数组、线段树、Treap 和 Trie。

README 提供中文名到英文文件名的索引，并明确表示代码可以自由使用，但不保证正确性。

### 一个文件就是一个完整程序

103 个 `.cpp` 文件全部包含 `main()`。它们通常将数据结构、算法、输入和输出写在同一个翻译单元中：

```text
stdin
→ 固定容量全局数组
→ 算法过程
→ stdout
```

这种形式便于竞赛时复制和修改，不适合通过头文件、命名空间和稳定接口复用。

### 缺少可验证资料

仓库没有：

- 每个算法的输入输出说明。
- 时间与空间复杂度。
- 示例输入输出。
- 单元测试或随机对拍。
- 构建脚本和 CI。
- 正确性证明或在线评测链接。

README 还说明索引可能滞后于实际代码。

## 我的判断

### 它是“模板参考”，不是“算法标准答案”

仓库最大的价值是用短代码展示竞赛实现形态。遇到某个经典名称时，可以快速找到数组布局、循环顺序和基本操作。但不能因为文件名正确就假设实现正确，原因包括：

- 输入约束和前置条件没有文档。
- 固定数组容量可能与目标问题不匹配。
- 有符号整数和 `INF` 可能溢出。
- 递归图算法可能栈溢出。
- 指针结构的资源所有权不完整。
- 某些文件是特定题目解法，不是通用实现。

真正复用前必须重新定义契约并用目标场景测试。

### 现代兼容性已经出现明确问题

全量 C++14 语法检查中：

- KMP 使用不安全且已移除的 `gets()`。
- 无旋 Treap 缺少 `strcmp()` 所需的 `<cstring>`。

其余 101 个文件语法通过，只能证明编译器能解析，不能证明结果、复杂度和边界行为正确。2019 年之后无更新也意味着现代编译器告警和标准库变化没有持续处理。

### 对工程项目的迁移成本高

这些实现缺少参数化容器、错误状态、迭代器、内存边界和测试接口。迁移到工程代码时，不应直接复制整个文件，而应：

1. 明确算法输入、输出、约束和失败语义。
2. 将固定宏替换为配置或按输入分配的容器。
3. 把算法逻辑从 `main()` 和 I/O 中分离。
4. 使用项目已有的数据结构和命名规范。
5. 补充最小、随机、极限和非法输入测试。
6. 与标准库或成熟实现交叉验证。

### 与 CppRobotics 的区别

CppRobotics 面向机器人定位、规划和控制，并提供 OpenCV 可视化；本仓库面向通用竞赛算法，主要通过标准输入输出运行。两者都是教学型单文件实现，也都不应直接视为生产库。

## 可复用内容

### 推荐用途

- 查找经典算法的数组式实现骨架。
- 对比同一问题的不同算法，例如最大流和最短路。
- 学习竞赛数据结构的核心操作。
- 为自己的算法库编写测试用例和接口时提供参考。
- 练习从“可提交程序”重构为“可复用组件”。

### 阅读顺序

1. 排序、二分、栈、队列和并查集。
2. BFS、DFS、最短路和最小生成树。
3. 树状数组、线段树、稀疏表。
4. KMP、Trie、AC 自动机。
5. Tarjan、最大流和二分图匹配。
6. Splay、Treap、可持久化结构和 Link-Cut Tree。
7. FFT、NTT、Miller-Rabin 和原根。

### 验证模板

对每个准备使用的算法至少检查：

```text
接口与索引范围
→ 最小和空输入
→ 随机小规模对拍
→ 最大规模性能
→ 溢出和递归深度
→ 重复元素、重边、自环或不连通情况
```
