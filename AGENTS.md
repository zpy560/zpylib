# 项目规则

## 项目目标

本项目是长期自用的本地 AI 知识编译系统。v1 只跑通：

```text
链接 / 文字型 PDF / 想法
→ 原料
→ 单源笔记
→ 标签与关联
→ 主题页
→ 问答 / 查找 / 选题
```

技术边界以 `techarc.md` 为准。修改实现边界前，先更新该文档。

## 目录约定

- `inbox/`：等待处理的链接、PDF 和想法。
- `raw/`：原始资料和提取文本。只追加，不覆盖。
- `notes/`：一项来源对应一篇单源笔记。
- `wiki/`：主题页和实体页，可随新资料重新编译。
- `briefs/`：基于主题页生成的选题。
- `indexes/`：生成式导航和索引。
- `templates/`：Markdown 模板、标签词表和数据约定。
- `scripts/`：确定性机械任务；v1 流程稳定前不新增脚本。
- `tests/`：脚本测试。
- `.agents/skills/`：本项目专用 Codex Skill。

不要在项目根目录外再创建 `fuxi/` 包装目录。

## 数据规则

1. Markdown 是主数据，不使用数据库或向量库。
2. `raw/` 中的原料是事实来源，禁止摘要覆盖原文。
3. 文件名使用 `YYYY-MM-DD-slug.md`；slug 使用小写英文和连字符。
4. 笔记元数据必须符合 `templates/frontmatter-schema.md`。
5. `tags` 只能从 `templates/tags.yaml` 中选择。
6. `related` 使用相对路径；建立关联时同步维护双向关系。
7. `id` 创建后不得修改。
8. 不把密钥、隐私资料或受保密约束的内容发送给 Codex。

## 工作流

### 入库

1. 先把输入保存到 `inbox/`。
2. 把原始文件或正文保存到 `raw/`。
3. 使用模板生成 `notes/` 单源笔记。
4. 从固定词表选择标签，不临时发明标签。
5. 查找关联笔记，更新双方的 `related`。
6. 更新相关 `wiki/` 页面和 `indexes/INDEX.md`。
7. 完成后报告新增和修改的文件。

### 问答

按以下顺序读取，够用即停：

1. `indexes/INDEX.md`
2. `wiki/`
3. `notes/`
4. `raw/`

回答需要区分原文事实、笔记判断和 AI 推断。

### 选题

选题写入 `briefs/`，必须包含标题、核心观点、来源、资料缺口。

## 修改与安全

- 修改 `wiki/` 前先确认当前版本已进入 Git 历史；如果仓库尚无基线提交，先提醒用户。
- 删除文件、移动原料、Git 回滚、安装依赖前必须获得用户确认。
- Git 提交统一使用 `zpy <zengpiaoyang123@163.com>` 身份。
- 抓取失败或遇到反爬时停止自动尝试，改为提示用户手动保存。
- 文字型 PDF 属于 v1；扫描件和图片型 PDF 不做 OCR。
- 周报属于 v2，不添加定时任务。
- 不开发独立 UI，不引入 Docker、Web 服务或 RAG 框架。

## 验证

每次变更后至少检查：

1. 文件是否位于正确目录。
2. frontmatter 是否完整。
3. 标签是否来自固定词表。
4. `related` 是否为有效相对路径且双向一致。
5. 索引是否需要更新。
6. 是否意外修改了 `raw/` 原料。
