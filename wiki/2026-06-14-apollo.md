## 当前判断

Apollo 是研究完整自动驾驶闭环和量产型规划控制工程组织方式的高价值参考。其主要优势是模块边界完整、道路场景表达明确、规划控制算法可插拔，并把轨迹 fallback、控制急停和系统安全监控分布在多个层级。

主要风险是配置面很大，源码中存在的插件或安全模块不代表部署时已启用。分析和二次开发必须把代码、DAG、pipeline、gflags、Topic 和车辆配置作为一个整体。

## 架构总览

```text
Drivers
→ Localization / Perception
→ Prediction
→ Planning
→ Control
→ Guardian
→ Canbus
→ Vehicle
```

配套系统：

- Routing 与 External Command：任务和路线入口。
- HD Map 与 PncMap：道路拓扑和参考线。
- Monitor：模块健康与功能安全状态。
- Dreamview Plus：调试、可视化和资源管理。
- Cyber RT：组件运行、通信、定时器和插件管理。

## 规划架构

```text
PlanningComponent
→ OnLanePlanning / NaviPlanning
→ PublicRoadPlanner
→ ScenarioManager
→ Scenario
→ Stage
→ Task
→ ADCTrajectory
```

核心数据对象：

- `LocalView`：规划输入聚合。
- `Frame`：单周期规划上下文。
- `ReferenceLineInfo`：候选参考线、路径、速度和决策。
- `PlanningContext`：跨周期场景状态。
- `DependencyInjector`：共享服务和状态依赖。

## 控制架构

```text
ADCTrajectory + Localization + Chassis
→ 输入与时间戳检查
→ ControlTask 插件流水线
→ ControlCommand
```

默认控制器为 LQR 横向控制和 PID 纵向控制，也提供 MPC 和扩展控制任务。控制支持单组件模式和预处理、控制核心、后处理子模块模式。

## 安全与兜底

- Planning：停止轨迹、task fallback、可选 EStop。
- Control：输入检查、超时检查、挡位保护、软急停。
- Monitor：关键组件健康检查和安全模式。
- Guardian：控制命令旁路或制动覆盖。
- Canbus：命令超时和底盘通信检查。

当前本地默认配置中 Canbus 不接收 Guardian，Guardian 也临时忽略超声波判断，不能把模块存在等同于安全链已启用。

## 已识别静态风险

1. `version.json` 为 9.0.0，与 11.0 系列 Git 标签关系不一致。
2. 控制子模块预处理对轨迹点按值遍历，修正不会写回。
3. Guardian 强制清除超声波故障和障碍物判断结果。
4. 安全行为依赖部署配置，需要逐项审计真实运行链。

## 对规划控制的启发

1. 按组件、业务状态和算法任务拆分变化频率。
2. 用统一 Frame 承载单周期数据，用 Context 承载跨周期状态。
3. 把场景切换、阶段迁移和数值优化分层。
4. 规划失败、控制失败和系统失败采用不同层级的兜底。
5. 插件化必须配套配置审计、加载验证和回放测试。
6. 安全评审应从最终执行命令反向追踪实际启用链。
